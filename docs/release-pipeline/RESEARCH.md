# Release Pipeline Research

## Overview

Build a complete release pipeline for Rerun so shipping new versions is a single command: notarized DMG, auto-updates via Sparkle, GitHub releases with changelogs, and a website download page. Modeled on the proven Chops and Clearly pipelines, adapted for Rerun's SPM-based build system.

## Problem Statement

Rerun has zero release infrastructure. There's no way to ship the app to users. Specifically:

- No notarization — macOS Gatekeeper will block the app
- No DMG — no drag-to-install experience
- No auto-update mechanism — users would have to manually download every version
- No download page — website only has a waitlist
- No changelog tracking
- No release automation — everything would be manual
- No version management beyond a hardcoded string

The GTM strategy targets a coordinated launch across HN, Product Hunt, Twitter, and GitHub. All of these require a working download link to a notarized app that users can install and that updates itself.

### Build pipeline detail

Rerun uses **Swift Package Manager**, not Xcode projects. This is the key architectural difference from the reference apps (Chops and Clearly both use xcodegen → xcodebuild). The build flow is:

```
swift build -c release
  → .build/release/rerun-daemon (binary)
  → bundle.sh assembles .app bundle:
      - Copies binary to Contents/MacOS/
      - Generates Info.plist with version, bundle ID, LSUIElement
      - Signs with Developer ID
```

The bundle has two variants:
- **Rerun.app** (`com.rerun.app`) — production
- **RerunDev.app** (`com.rerun.dev`) — dev, with profile isolation

### What's missing from bundle.sh for release

1. **Hardened runtime** — `codesign` currently lacks `--options runtime`, required for notarization
2. **MLX metallib** — Metal shader compilation exists in `dev.sh` but not in `bundle.sh` (release builds won't have GPU acceleration)
3. **Sparkle framework** — no framework embedding, no Info.plist keys
4. **Entitlements** — may be needed if hardened runtime breaks MLX JIT (to be tested)

## Reference Analysis: Chops & Clearly

Both apps are by the same developer (same team ID, same signing identity) with nearly identical release pipelines. They represent the target architecture.

### Release script flow

Both use `scripts/release.sh <version>`:

```
1. Load .env (APPLE_TEAM_ID, APPLE_ID, SIGNING_IDENTITY_NAME)
2. Validate notarytool keychain profile "AC_PASSWORD"
3. Build (xcodegen → xcodebuild archive → export)
4. Create DMG (hdiutil)
5. Notarize (xcrun notarytool submit --wait)
6. Staple (xcrun stapler staple on app + DMG)
7. Git tag + push
8. Generate Sparkle appcast (sign_update → XML)
9. GitHub release (gh release create with DMG + changelog)
```

**Key adaptation needed:** Step 3 uses xcodebuild archive/export. Rerun must use `swift build -c release` + `bundle.sh` instead. Everything else is directly reusable.

### DMG creation

- **Chops:** Fancy — writable DMG, mounts, adds Applications symlink + background image, AppleScript for Finder icon layout, converts to compressed UDZO
- **Clearly:** Simple — `hdiutil create -srcfolder app -ov -format UDZO`

**Recommendation:** Start with Clearly's simple approach. A fancy DMG can come later.

### Sparkle integration

Both apps use Sparkle 2 via SPM (`^2.6.0`):

- **Info.plist keys:** `SUFeedURL` (appcast URL), `SUPublicEDKey` (EdDSA public key)
- **EdDSA key pair:** Generated once via Sparkle's `generate_keys` tool, private key stored in macOS Keychain
- **Updater init:** `SPUStandardUpdaterController(startingUpdater: true, updaterDelegate: nil, userDriverDelegate: nil)` — starts automatic periodic checks
- **Menu item:** "Check for Updates…" targeting the updater controller
- **Appcast:** XML file served from website, generated by release script using `sign_update` to get EdDSA signature for the DMG

**Appcast hosting:**
- Chops: `https://chops.md/appcast.xml` (Astro site on Cloudflare)
- Clearly: `https://clearly.md/appcast.xml` (static site)
- Rerun: `https://rerun.md/appcast.xml` (Astro site on Vercel) — served from `website/public/appcast.xml`

**Download URLs:** Both point to GitHub releases (`https://github.com/Shpigford/{app}/releases/download/v{VERSION}/{App}.dmg`). This is smart — GitHub handles CDN/bandwidth, no self-hosting needed.

### /release skill

Both have `.claude/skills/release/SKILL.md` with identical structure:

1. Verify prerequisites (.env, notarytool, clean tree, main branch)
2. Determine version from git history (semver from conventional commits)
3. Confirm version with user
4. Update CHANGELOG.md (rewrite commits as user-facing entries, confirm with user)
5. Update marketing site version string
6. Commit version bump
7. Run `./scripts/release.sh <VERSION>`
8. Push and report GitHub release URL

**Key design:** The skill handles the human-judgment parts (version decision, changelog writing, confirmation). The script handles the mechanical parts (build, sign, notarize, package, upload). Clean separation.

### Credential management

Both use identical patterns:
- `.env` file (gitignored) with `APPLE_TEAM_ID`, `APPLE_ID`, `SIGNING_IDENTITY_NAME`
- `.env.example` (committed) with placeholder values
- `AC_PASSWORD` keychain profile for notarytool (created once via `xcrun notarytool store-credentials`)
- EdDSA private key in Keychain (managed by Sparkle's `generate_keys`)

## Recommended Approach

### Build adaptation for SPM

The release script calls `bundle.sh` instead of xcodebuild:

```bash
cd app && VERSION="$VERSION" CODESIGN_IDENTITY="$SIGNING_IDENTITY" ./bundle.sh prod
```

`bundle.sh` already respects `VERSION` and `CODESIGN_IDENTITY` as env vars. The release script constructs the signing identity string from `.env` values and passes it in.

### Hardened runtime

Add `--options runtime` to the codesign call in `bundle.sh`. This enables hardened runtime, which Apple requires for notarization.

**Risk:** MLX uses Metal compute shaders and possibly JIT compilation. Hardened runtime restricts JIT by default. If the app crashes after enabling hardened runtime, an entitlements file with `com.apple.security.cs.allow-jit` and/or `com.apple.security.cs.allow-unsigned-executable-memory` will be needed. This must be tested empirically — build with hardened runtime, run the app, exercise the chat feature (which uses MLX).

### Sparkle + SPM

This is the trickiest integration. When Sparkle is added as an SPM dependency, `swift build` downloads it as a binary xcframework artifact. The built daemon links against it dynamically. But SPM does not create `.app` bundles or embed frameworks — that's `bundle.sh`'s job.

`bundle.sh` must:
1. Find `Sparkle.framework` in `.build/artifacts/sparkle/Sparkle/`
2. Copy it to `Contents/Frameworks/`
3. Sign it with the same Developer ID
4. Ensure rpaths are correct so the binary finds the framework at runtime

This should only happen for the production bundle (`com.rerun.app`), not RerunDev.

### Notarization

Standard Apple notarization flow, identical to Chops/Clearly:

```bash
xcrun notarytool submit Rerun.dmg --keychain-profile "AC_PASSWORD" --wait
xcrun stapler staple Rerun.app
# Re-create DMG with stapled app
xcrun stapler staple Rerun.dmg
```

One-time setup (not scriptable, documented in .env.example):
```bash
xcrun notarytool store-credentials "AC_PASSWORD" \
  --apple-id "$APPLE_ID" \
  --team-id "$APPLE_TEAM_ID" \
  --password "<app-specific-password-from-appleid.apple.com>"
```

### DMG naming

Use `Rerun.dmg` (not versioned). This lets GitHub's `latest/download/Rerun.dmg` URL work as a stable download link that always points to the latest version. The version is embedded in the app bundle and appcast, not the filename.

### Version management

Keep `app/Sources/RerunCore/Rerun.swift` as the single source of truth. The release script/skill updates it along with three other references:
- `app/bundle.sh` default `VERSION` value
- `app/dev.sh` Info.plist heredoc
- `app/Tests/RerunCoreTests/RerunCoreTests.swift` assertion

### Website download experience

Replace or supplement the waitlist with a direct download button linking to `https://github.com/usererun/rerun/releases/latest/download/Rerun.dmg`. Display the current version and macOS requirement. A small JSON data file (`website/src/data/version.json`) gets updated by the release process.

## Security Considerations

**This is a public, open-source repo. Zero secrets in source.**

| Secret | Where it lives | Safe to commit? |
|--------|---------------|-----------------|
| Apple ID email | `.env` (gitignored) | No |
| App-specific password | macOS Keychain (`AC_PASSWORD` profile) | No |
| Signing identity name | `.env` (gitignored) + hardcoded in bundle.sh | The bundle.sh default is already public — acceptable |
| Team ID | `.env` (gitignored) + hardcoded in bundle.sh | Already public (in every signed binary) — acceptable |
| EdDSA private key | macOS Keychain (Sparkle manages) | **Never** |
| EdDSA public key | Info.plist in bundle.sh | Yes — it's public by design |
| Appcast XML | `website/public/appcast.xml` | Yes — contains signatures, not keys |
| `.env.example` | Committed | Yes — placeholder values only |

**Rules:**
- `.env.example` uses only placeholders: `YOUR_TEAM_ID`, `your-apple-id@example.com`, `Your Name or Company`
- Release script reads all secrets from `.env` or Keychain, contains zero secrets itself
- EdDSA key generation is a one-time manual step, private key stays in Keychain forever
- Never `git add .env` — it's already gitignored

## Phased Implementation

### Phase 1: Release Infrastructure Foundation
- `.env.example` with placeholder values
- `CHANGELOG.md` with Keep a Changelog format
- Hardened runtime in `bundle.sh` (`--options runtime`)
- MLX metallib compilation in `bundle.sh` (currently only in dev.sh)
- Test hardened runtime doesn't break MLX

### Phase 2: Release Script
- `scripts/release.sh` — build → DMG → notarize → staple → tag → GitHub release
- Adapted from Chops/Clearly pattern using SPM build instead of xcodebuild
- Version string updates across all four files
- Changelog extraction for release notes

### Phase 3: Sparkle Auto-Update
- Add Sparkle 2 SPM dependency to Package.swift (RerunDaemon target only)
- Generate EdDSA key pair (one-time manual step)
- Embed Sparkle.framework in production bundle
- Add SUFeedURL and SUPublicEDKey to generated Info.plist (production only)
- SPUStandardUpdaterController in daemon startup
- "Check for Updates…" menu item in StatusBarController
- Appcast generation in release script

### Phase 4: Download Experience
- Download button on website replacing/supplementing waitlist
- Version display from data file
- Release process updates website version

### Phase 5: /release Skill
- `.claude/skills/release/SKILL.md`
- Prerequisites check, semver from commits, changelog management, version bump, script execution, push and report

### Phase order

```
Phase 1 (foundation) → Phase 2 (release script) → Phase 3 (Sparkle) ┐
                                                  → Phase 4 (website) ┤
                                                  → Phase 5 (skill)   ┘
```

Phases 3, 4, 5 can proceed in parallel after Phase 2.

## Open Questions

1. **Domain:** Is the website domain `rerun.md` or something else? The appcast URL depends on this.
2. **MLX + hardened runtime:** Will enabling hardened runtime break MLX JIT? Needs empirical testing in Phase 1.
3. **Waitlist fate:** Replace entirely with download, or keep as secondary CTA for pre-launch?
4. **DMG styling:** Start with simple UDZO (recommended) or invest in a styled DMG with background image upfront?
5. **Homebrew Cask:** The GTM strategy mentions `brew install --cask rerun` for launch day. Should this be part of this initiative or a follow-up?
